# 双目上传分组与音频录制设计方案

## 1. 背景

当前应用已经具备以下基础能力：

- iOS 端可完成单次录制，并在会话目录下生成 `data.mov`、`data.jsonl`、`calibration.json`、`metadata.json`、`frames2/` 等文件。
- Flutter 端在停止录制后会立即自动入上传队列。
- 上传请求当前只显式携带 `file`、`sessionName`、`sessionPath`。
- 后端已经支持接收 `captureName`、`sceneName`、`seqName`，但客户端还没有对应的用户输入与传递链路。
- 后端当前会在落盘后复制一份 ZIP，生成镜像文件 `(1).zip`，用于“模拟双目”。

本方案的目标是先梳理用户侧逻辑和数据流，不直接改代码。

## 2. 当前问题

### 2.1 双目识别方式不正确

当前“模拟双目”的方式是服务端在收到一个 ZIP 后，再复制出一份镜像 ZIP。这个方案有几个问题：

- 复制出来的两个 ZIP 实际是同一份数据，不能真实表达“两个同时拍摄的视频”。
- 服务端无法知道哪两个录制会话是真正同时拍摄的，只能假设一个会话被当作双目。
- 后续算法如果要区分真实双目、单目、纯场景、带人场景，当前镜像策略会误导处理流程。

### 2.2 上传分组信息没有用户入口

后端已经有 `captureName`、`sceneName`、`seqName` 这三个归档维度，但客户端目前没有：

- 录制前的归档策略选择；
- 上传前的确认与修正入口；
- “将两个录制标成同一次同时拍摄”的显式分组能力。

### 2.3 当前自动上传时机过早

停止录制后立即自动上传虽然省操作，但对于本次需求有两个明显问题：

- 用户还没来得及选择本次录制是“纯场景”还是“带人”。
- 用户还没来得及指定这次录制要归到哪个 `scene`、哪个 `seq`，以及是否与另一条视频配对。

### 2.4 录制没有音频

当前录制只采集视频和传感器，不采集麦克风音频，导致：

- 上传数据无法支持带音频的后处理；
- 服务端也无法区分“本次数据本来就没有音频”和“本次应有音频但录制失败”。

## 3. 设计目标

- 移除服务端“复制 ZIP 模拟双目”的策略。
- 让用户在录制相关流程中，明确表达本次上传的业务归档信息。
- 让服务端能够准确知道哪两个会话是同时拍摄的一组。
- 让服务端能够区分“纯场景”与“带人场景”。
- 为新录制增加音频采集，并把音频是否存在写入元数据。
- 对旧录制保持兼容，避免因为没有新增字段而无法上传。

## 4. 设计原则

- 机器生成的数据和用户填写的业务标签分离存储。
- 录制完成后先形成“待上传会话”，用户确认业务信息后再真正入上传队列。
- 默认尽量少打断用户，但关键业务字段必须在上传前补齐。
- 所有新增业务字段都要同时满足两种用途：
  - 本地可回显、可编辑、可重试；
  - 服务端可直接消费，不依赖再解析目录名猜测。

## 5. 核心方案概述

本次方案将上传链路分为两层元数据：

- 采集事实元数据：描述设备、录制模式、是否有音频、会话自身的技术事实。建议继续写入 `metadata.json`，或者在其内部新增字段。
- 上传业务元数据：描述这次上传属于哪个 `scene` / `seq`，是否纯场景，是否属于双目配对，配对 ID 是什么。建议新增独立文件 `upload_context.json`。

新的整体流程如下：

1. 用户开始录制前，选择或沿用一个“录制预设”。
2. 原生侧按预设执行录制，并在本地写入基础采集元数据。
3. 停止录制后，不直接自动上传，而是弹出“上传确认面板”。
4. 用户确认本次上传的 `scene` / `seq` / 场景类型 / 双目分组信息。
5. 客户端将这些字段写入 `upload_context.json`，然后再加入上传队列。
6. 上传请求除 ZIP 外，同时显式携带业务字段。
7. 服务端只保存一份 ZIP，不再复制镜像 ZIP；双目关系由业务字段判断。

## 6. 用户侧交互设计

### 6.1 新增录制预设

建议在 Home 页增加一个轻量级“录制预设”入口，位于录制按钮附近或顶部状态区域。预设至少包含以下字段：

- `captureType`
  - `scene_only`：纯场景
  - `human_in_scene`：带人
- `audioPolicy`
  - `required`：需要录制音频
  - `optional`：尽量录制音频，失败时允许无音频继续
- `pairingMode`
  - `single`：普通单次上传
  - `stereo_group`：本次录制将参与双目配对
- `archiveStrategy`
  - `new_scene_new_seq`
  - `existing_scene_new_seq`
  - `existing_scene_existing_seq`

说明：

- 这里的“录制预设”是为了减少重复输入，不要求用户在录制前把所有文本都填完。
- 文本型字段如 `sceneName`、`seqName`、`stereoGroupId` 可以在上传确认时再填写或修正。
- 预设需要本地持久化，默认沿用上一次选择。

### 6.2 停止录制后的上传确认面板

当前停止录制后会直接 `enqueueSession(sessionPath)`。本方案改为先弹出一个上传确认面板，再决定是否入队。

面板建议包含：

- 场景类型
  - 纯场景
  - 带人
- Scene 策略
  - 新建场景
  - 使用最近场景
  - 手动输入场景名
- Seq 策略
  - 新建 seq
  - 加入已有 seq
  - 手动输入 seq 名
- 双目分组
  - 不参与双目
  - 创建一个新的双目组
  - 加入一个已有双目组
- 音频状态提示
  - 已录制音频
  - 本次无音频
  - 麦克风权限被拒绝

确认按钮行为：

- “确认并上传”：写入 `upload_context.json`，再入上传队列。
- “仅保存，不上传”：保留会话目录，后续在文件浏览页再补传。
- “取消”：回到主页，不丢失录制结果。

### 6.3 文件浏览页的手动补传入口

`RecordingsBrowserPage` 当前只有“上传/重试上传”动作。建议改成：

- 若该会话没有完整的上传业务元数据，点击“上传”时先弹出上传确认面板。
- 若该会话已有 `upload_context.json`，则预填表单，允许用户修改后再上传。
- 失败重试时默认沿用上一次业务元数据，但仍允许修改。

这样可以兼容以下情况：

- 用户录完后暂时不想上传；
- 用户要回头把两段视频配成同一组；
- 旧录制没有新增字段，需要在首次上传前补信息。

## 7. 双目分组设计

### 7.1 为什么不能再靠复制 ZIP

复制 ZIP 只能制造两个文件名，不能提供真实语义。真实需要表达的是：

- 哪些会话属于同一次同时拍摄；
- 这组内有几条录制；
- 这组拍摄是否带人；
- 是否包含音频。

因此双目关系必须从“文件复制”切换为“显式分组元数据”。

### 7.2 建议的数据字段

建议在 `upload_context.json` 与 multipart 字段中同时携带以下信息：

- `captureType`
  - `scene_only`
  - `human_in_scene`
- `pairGroupId`
  - 一个稳定字符串，用于标记“同时拍摄的一组”
  - 例如 `stereo_20260414_153012_ab12`
- `sceneName`
- `seqName`
- `captureName`

其中：

- `pairGroupId` 是最关键字段，服务端据此判断两个会话是同组。
- 如果一条录制不参与双目，则 `pairGroupId` 为空。

### 7.3 扫码建组与共享码规则

如果双目视频由两个不同用户、两台不同设备同时拍摄，那么只共享 `pairGroupId` 并不足以保证它们落到服务端同一个目录下。

原因是：

- 服务端当前真正决定落盘目录的是 `captureName`、`sceneName`、`seqName`；
- `pairGroupId` 只负责表达“这两个会话属于同一组”，本身不参与目录路由；
- 因此如果两个用户只约定了同一个 `pairGroupId`，但各自填写了不同的 `sceneName` 或 `seqName`，两条视频仍会被上传到不同目录。

为了解决这个问题，建议把“建组共享”的最小单位定义为一份完整的组配置，而不是单独一个组号。

组配置至少包含：

- `captureName`
- `captureType`
- `sceneName`
- `seqName`
- `pairGroupId`

推荐交互如下：

1. A 用户选择“创建新的双目组”。
2. 客户端自动生成一份组配置：
   - `sceneName = scene_YYYYMMDD_HHMMSS`
   - `seqName = seq0`
   - `pairGroupId = group_YYYYMMDD_HHMMSS`
3. A 用户可通过两种方式把这份组配置分享给 B：
   - 显示二维码，供 B 扫码加入
   - 显示共享码，供 B 手动输入
4. B 用户加入时，不只是得到 `pairGroupId`，而是完整导入同一份 `captureName` / `sceneName` / `seqName` / `pairGroupId`。
5. 两边上传时都带同一组字段，服务端自然会把它们写入同一个 `scene/seq` 目录下。

也就是说：

- “同目录”依赖 `captureName + sceneName + seqName`
- “同组关系”依赖 `pairGroupId`
- 扫码或输入共享码时，必须同步这四个字段，而不是只同步 `pairGroupId`

共享码建议承载完整组配置，而不是只承载一个纯文本组号。这样即使用户不扫码，也可以通过复制粘贴共享码保证目录一致。

客户端需要支持两种加入方式：

- 手动输入组号
  - 仅适合已经明确知道本次 `scene/seq` 应如何填写的高级用户
  - 只能保证“同组”，不能自动保证落到同一目录
- 输入共享码或扫码导入
  - 自动同步 `captureType`、`sceneName`、`seqName`、`pairGroupId`
  - 可以保证服务端落到同一个 `scene/seq` 目录

### 7.4 Scene / Seq 与双目分组的关系

推荐定义如下：

- `sceneName`：表示场景级别，通常一个地点或一段布置共享同一场景名。
- `seqName`：表示一次具体采集序列。
- `pairGroupId`：表示“同时拍摄的一组”。

也就是说：

- 两条同时拍摄的视频必须共享同一个 `pairGroupId`。
- 它们通常也应该共享同一个 `sceneName` 与 `seqName`。
- 但真正的“同时拍摄关系”不依赖目录名推断，而以 `pairGroupId` 为准。
- 如果要保证服务端落到同一个目录，就必须同步 `captureName`、`sceneName`、`seqName`，不能只同步 `pairGroupId`。

这样做的好处是：

- 归档结构仍然兼容现有 `capture/scene/seq` 目录体系；
- 双目关系不再依赖复制 ZIP；
- 后处理可以先按 `pairGroupId` 聚合，再按 `sceneName` / `seqName` 校验。

## 8. Scene / Seq 策略设计

### 8.1 推荐保留后端已有字段

后端已经支持：

- `captureName`
- `sceneName`
- `seqName`

客户端应直接补齐这三项，不新增另一套完全独立的命名体系。

### 8.2 客户端策略表达

建议把用户选择表达为“策略 + 最终值”两部分：

- 策略字段
  - `sceneStrategy`
  - `seqStrategy`
- 最终字段
  - `sceneName`
  - `seqName`

原因是：

- 上传时真正落到服务端的还是最终值；
- 本地保留策略字段，便于以后做自动命名和 UI 回显。

### 8.3 推荐策略

- `sceneStrategy`
  - `new_scene`
  - `reuse_recent_scene`
  - `manual_scene`
- `seqStrategy`
  - `new_seq`
  - `reuse_recent_seq`
  - `manual_seq`

推荐默认行为：

- 新地点或新布置：`new_scene + new_seq`
- 同一地点继续拍：`reuse_recent_scene + new_seq`
- 两条同时视频：第二条通常 `reuse_recent_scene + reuse_recent_seq`，并加入相同 `pairGroupId`

## 9. 音频录制设计

### 9.1 用户侧目标

从用户视角，本次改造要达到以下效果：

- 开始录制时能明确知道本次是否会录音；
- 如果麦克风权限有问题，用户能收到清晰反馈；
- 上传后服务端能知道该会话是否包含音轨。

### 9.2 权限交互

建议在开始录制前同时检查相机和麦克风权限。

推荐交互：

- 相机权限未授权：保持当前逻辑，阻止录制。
- 麦克风权限未授权：
  - 推荐弹窗给两个选项：
    - 去设置/授权后录制
    - 本次继续无音频

这样兼顾两点：

- 新需求默认鼓励带音频；
- 不会因为麦克风权限问题把所有录制都彻底阻塞。

### 9.3 录制事实字段

建议写入 `metadata.json` 的字段：

- `audio_enabled`
- `audio_permission_granted`
- `audio_track_expected`
- `audio_track_present`
- `audio_sample_rate`（若可获得）
- `audio_channel_count`（若可获得）

字段含义区分如下：

- `audio_track_expected`：本次录制是否计划带音频。
- `audio_track_present`：最终文件里是否真的有音轨。

这能区分以下几种情况：

- 用户明确选择不录音；
- 用户想录音，但权限被拒；
- 用户想录音，也录成功了；
- 用户想录音，但中途写入失败。

### 9.4 UI 提示

录制中建议增加一个很轻量的音频状态标识：

- 麦克风正常：显示麦克风图标
- 无麦克风权限：显示禁用图标
- 当前静默录制：显示“无音频”

停止录制后的上传确认面板也要回显该状态。

## 10. 本地数据结构设计

## 10.1 `metadata.json`

建议继续保留为“采集事实元数据”，新增但不删除现有字段。

建议新增结构示意：

```json
{
  "device_model": "iPhone16,2",
  "platform": "ios",
  "capture_mode": "depthAndWide",
  "depth_mode_required": true,
  "audio_enabled": true,
  "audio_permission_granted": true,
  "audio_track_expected": true,
  "audio_track_present": true
}
```

## 10.2 `upload_context.json`

建议新增一个用户业务元数据文件，专门描述上传意图。

结构示意：

```json
{
  "version": 1,
  "captureName": "ios_capture",
  "captureType": "human_in_scene",
  "sceneStrategy": "reuse_recent_scene",
  "sceneName": "office_demo_01",
  "seqStrategy": "reuse_recent_seq",
  "seqName": "seq_0007",
  "pairGroupId": "stereo_20260414_153012_ab12",
  "shareCode": "sdrgrp1_xxx",
  "confirmedAt": "2026-04-14T15:32:10Z"
}
```

说明：

- `upload_context.json` 允许在录制后被补写或编辑。
- 上传队列以它作为业务字段来源，而不是只靠 UI 临时状态。
- 文件要一并打进 ZIP，方便服务端追溯。

## 11. 上传协议设计

### 11.1 multipart 字段

建议上传时显式携带：

- `file`
- `sessionName`
- `sessionPath`
- `captureName`
- `sceneName`
- `seqName`
- `captureType`
- `pairGroupId`
- `audioTrackPresent`

### 11.2 服务端处理原则

服务端改造原则如下：

- 保留现有 `captureName` / `sceneName` / `seqName` 目录逻辑；
- 移除 `copyFinalToMirror()` 及其调用；
- 上传成功后只保留一份原始 ZIP；
- 在服务端记录中持久化新增业务字段；
- 以后双目处理按 `pairGroupId` 聚合，不再根据 `(1).zip` 推断。

## 12. 对后端的配合要求

虽然这次文档重点是用户侧逻辑，但有几个服务端点必须同步，否则客户端改造后效果不完整。

### 12.1 删除镜像复制

必须移除：

- 上传落盘后的镜像 ZIP 复制
- `(1).zip` 这套命名约定

因为它会与真实配对关系冲突。

### 12.2 `data2.mov` 维持非必需

当前后端在解压场景文件时会忽略或删除 `data2.mov`。结合当前产品决策，后端后处理不需要直接消费 `data2.mov`，因此这部分逻辑可以暂时维持不变。

这里要强调的是：

- 双目“同组”关系由 `pairGroupId` 表达，而不是由 `data2.mov` 是否存在表达；
- 即使后端不消费 `data2.mov`，也不能再用复制 ZIP 代替真实双目关系。

### 12.3 服务端记录建议

建议服务端在上传成功记录中增加：

- `captureType`
- `pairGroupId`
- `audioTrackPresent`

这样后续离线处理和查询接口都可以直接使用。

## 13. 自动上传策略调整

当前是“停止即自动入队”。本方案建议改成“停止即弹确认，确认后入队”。

推荐逻辑：

- 如果本次会话的上传业务字段完整：
  - 默认弹出确认面板，用户一键确认即可上传
- 如果字段不完整：
  - 强制用户补齐后再上传
- 如果用户关闭面板：
  - 会话进入“仅本地保存”状态，不自动上传

这样既保留了原有高效流程，也满足新增标签与分组需求。

## 14. 旧数据兼容策略

### 14.1 旧录制目录

旧录制没有：

- `upload_context.json`
- 音频字段
- `pairGroupId`
- `captureType`

兼容方案：

- 首次上传旧录制时，强制打开上传确认面板；
- 用户补齐场景类型、Scene/Seq、是否双目分组；
- `audio_track_present` 对旧录制默认为 `false`；
- 服务端对缺失 `pairGroupId` 的记录按单目处理。

### 14.2 已上传历史数据

对于已经通过镜像 ZIP 方式上传的历史数据：

- 不在本次客户端方案内自动迁移；
- 后端后处理链路需要区分“历史镜像模式”和“新显式分组模式”；
- 可以通过记录时间或是否存在 `pairGroupId` 进行兼容判断。

## 15. 推荐实施顺序

### 阶段 1

- 客户端增加 `upload_context.json` 设计与 UI 流程
- 停止录制后改为先确认再入队
- 文件浏览页上传前补业务信息

### 阶段 2

- 上传协议补齐 `captureType`、`pairGroupId`、`audioTrackPresent`
- 后端持久化这些字段
- 后端移除 ZIP 镜像复制逻辑

### 阶段 3

- iOS 原生增加音频采集
- 补充麦克风权限流转与录制状态展示
- 在 `metadata.json` 中写入音频事实字段

### 阶段 4

- 评估后端是否需要保留 `data2.mov` 解压结果
- 根据后处理链路决定双路数据消费方式

## 16. 风险与注意事项

- 如果继续保留“停止即自动上传”而没有确认面板，新增业务字段很容易缺失或填错。
- 如果只在上传请求里携带业务字段，而不写入本地文件，重试和补传时会丢失上下文。
- 如果服务端删除了镜像复制，但客户端又没有提供 `pairGroupId`，后续就无法恢复真实双目关系。
- 如果增加音频后不处理麦克风权限，用户会以为录上了音，但实际上文件仍然无音频。
- 如果后续双目算法需要真实第二路视频，当前服务端对 `data2.mov` 的忽略策略必须同步调整。

## 17. 本方案最终结论

本次需求不应再沿用“上传后复制 ZIP 模拟双目”的旧方案，而应改为：

- 客户端显式采集并保存上传业务元数据；
- 通过 `pairGroupId` 明确表达哪两个会话是同时拍摄的一组；
- 通过 `captureType` 区分纯场景和带人场景；
- 通过音频元数据明确表达本次是否成功录音；
- 停止录制后先确认再上传；
- 服务端只保存一份 ZIP，不再做镜像复制。

这是后续支持真实双目处理、场景分类和音频处理的最小正确方案。
