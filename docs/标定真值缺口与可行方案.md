# 标定「真值」缺口与可行方案

本文说明：在 [`Recording参数来源说明.md`](Recording参数来源说明.md) 所描述的实现中，`calibration.json` 里有两类对 **几何重建 / 视觉-惯性管线** 至关重要的量，当前**仅有字段与占位数值，未能达到与物理光学、装配一致的真值**。文中归纳**未能按真值实现的原因**，以及**后续可能落地的实现路径**（需额外开发或标定，不在 P0「最小闭环」范围内）。

**与 P1 的关系（[`MVP开发优先级.md`](MVP开发优先级.md)）**：P1 已加入 **对焦/曝光/白平衡锁定**、**JSONL 按时间排序与缓冲落盘**，并开启 **相机内参矩阵附件** 能力（便于后续从 `CMSampleBuffer` 解析真实 `focalLength`/`principalPoint`）。这些措施能 **提高单段录制内参数稳定性** 与 **数据文件可复现性**，但 **不替代** 从系统标定 API 或离线标定得到的 **内参真值**；`imuToCamera` 仍为占位，**不因 P1 而改变**。

**关联文档**：[Recording采集与落盘对照表.md](Recording采集与落盘对照表.md)、[MVP开发优先级.md](MVP开发优先级.md)、[Spectacular_AI_DATA_FORMAT_中文.md](Spectacular_AI_DATA_FORMAT_中文.md)、[Spectacular样例对齐差距说明.md](Spectacular样例对齐差距说明.md)。

**与样例对齐后的内参**：若 `CMSampleBuffer` 携带 **`kCMSampleBufferAttachmentKey_CameraIntrinsicMatrix`** 且解析成功，`calibration.json` 与 JSONL 帧内 `calibration` 可使用 **系统给出的 fx、fy、cx、cy（像素）**，在针孔模型下可视为**内参主项的真值来源**；**畸变系数**与 **Kannala-Brandt 等模型** 仍属未接入项，见下文第二节、第三节。

---

## 一、当前占位、未达真值的两项


| 项目                           | `calibration.json` 中的现状                                           | 对下游的含义                          |
| ---------------------------- | ----------------------------------------------------------------- | ------------------------------- |
| **① 相机内参（焦距、主点、畸变）**         | **焦距/主点**：优先 **视频帧内参矩阵附件** 解析值；若附件缺失则 **经验比例 + 中心主点**。**畸变**：仍 **未** 写入与设备一致的系数或查找表 | 无畸变或模型不匹配时，大视场边缘与部分 VIO 仍受限  |
| **② IMU→相机外参 `imuToCamera`** | **4×4 单位矩阵** 占位                                                   | 惯性量与相机坐标系未对齐时，紧耦合视觉-惯性算法无法依赖该矩阵 |


二者在格式上均已存在，满足「文件可读、字段齐全」的 MVP 要求，但**数值语义**不等于设备真实标定。

---

## 二、① 真实相机内参未能按真值实现的原因

1. **P0 目标刻意收窄**
  MVP 优先验证 `**data.mov` + `data.jsonl` + 占位 `calibration.json`** 的闭环，避免在首版引入：分辨率与内参同步、对焦/变焦导致内参变化、畸变模型选型（如 Kannala-Brandt vs pinhole）等分支。
2. **「真值」来源不止一种**
  苹果在 **视频流、深度流、ARKit** 等路径上提供的标定数据形态不同（例如 3×3 内参矩阵、查表畸变、或内部已校正图像）。要在录制管线里统一写入 Spectacular 风格的 `focalLengthX/Y`、`principalPointX/Y`、`distortionCoefficients`，需要**显式选型、坐标与分辨率对齐**，这超出 P0 的占位策略。
3. **畸变尤其依赖采集模式**
  完整畸变（如 `AVCameraCalibrationData` 的查找表）常与 **深度、多摄标定交付** 等配置绑定；纯 `AVCaptureVideoDataOutput` 路径下，**能否**以及**以何种形式**拿到畸变，需按设备与会话配置核实，不能假设与样例 `recording_*` 完全一致。

---

## 三、① 可能实现的方案（内参 / 畸变）


| 方向              | 说明                                                                                                                                                                                                                                                 |
| --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **视频帧内参矩阵（优先）** | 在 `AVCaptureConnection` 上启用 `**isCameraIntrinsicMatrixDeliveryEnabled`**，从 `**CMSampleBuffer**` 读取附件 `**kCMSampleBufferAttachmentKey_CameraIntrinsicMatrix**`，得到 `**matrix_float3x3**`，换算为 `focalLengthX/Y`、`principalPointX/Y`（像素单位，须与当前编码分辨率一致）。 |
| **深度 / 标定数据路径** | 若产品允许开启相应采集能力，使用 `**AVCameraCalibrationData`** 获取 `**intrinsicMatrix**`、`**lensDistortionLookupTable**` 等，再映射到自有 JSON 字段或约定「查表畸变」的存储方式。                                                                                                            |
| **ARKit**       | `**ARCamera.intrinsics`** 提供针孔内参；畸变多在 ARKit 内部处理，是否导出与 Spectacular 一致的系数需单独设计。                                                                                                                                                                     |
| **离线标定**        | 张正友标定板等，在固定分辨率与对焦策略下拟合内参与畸变，作为会话级或机型级常量写入 `calibration.json`。                                                                                                                                                                                      |


**注意**：变焦、对焦、多分辨率切换会导致内参变化；录制期间应**锁定**与写入文件一致的分辨率与镜头配置，并在文档中说明有效条件。

---

## 四、② 真实 `imuToCamera` 未能按真值实现的原因

1. **无「一键读取工厂外参」的公开 API**
  **Core Motion** 提供 IMU 测量，**不**提供「IMU 坐标系到相机坐标系」的 4×4 齐次变换矩阵。苹果未将此类装配标定作为通用公共接口暴露给任意 `AVCaptureSession` 应用。
2. `**AVCameraCalibrationData.extrinsicMatrix` 语义不同**
  该类外参多描述相机相对**某一参考系**（例如多摄、深度对齐语境），**不能**直接等同于「IMU 刚体 → 相机光学中心」的通用 `imuToCamera`。
3. **P0 选择可复现的占位**
  单位阵在工程上便于发现「未标定」状态，避免使用随意假矩阵被误认为已标定。

---

## 五、② 可能实现的方案（IMU→相机外参）


| 方向                  | 说明                                                                                            |
| ------------------- | --------------------------------------------------------------------------------------------- |
| **离线手眼标定**          | 利用标定板、已知运动或专用标定流程，估计刚体变换 T_{\text{cam}\leftarrow\text{imu}}，写入 `calibration.json`；需固定机型与相机模块。 |
| **多帧联合估计**          | 在受控序列中联合优化外参与内参（需可观测性），属于算法/标定软件范畴，非单次 API 调用。                                                |
| **ARKit 与运动融合（间接）** | 通过 AR 相机位姿与设备运动在统一世界系下的关系**约束**外参，实现路径复杂，且与「纯 AVFoundation 录制」架构不同，需单独验证是否满足下游工具链输入假设。        |
| **文献 / 机型经验值**      | 极少数场景使用公开论文或近似值，风险高，仅作粗初始化，不适合标为「真值」。                                                         |


---

## 六、小结


| 缺口            | 未真值实现的主因                    | 推荐演进方向                                                     |
| ------------- | --------------------------- | ---------------------------------------------------------- |
| 畸变（及完整相机模型） | 当前仅 **针孔 + 附件 fx/fy/cx/cy**；无畸变系数/查表 | **AVCameraCalibrationData**、深度流或离线标定 |
| `imuToCamera` | 系统无直接 API                   | **离线标定**或联合估计；与 ARKit 融合需单独架构与验证                           |


实现任一路径时，请同步更新 [`Recording参数来源说明.md`](Recording参数来源说明.md) 与 [`Recording采集与落盘对照表.md`](Recording采集与落盘对照表.md)，标明字段为 **API 导出**、**离线标定** 或仍为 **占位**。内参主项若来自 **`CMSampleBuffer` 内参矩阵**，见 [`Spectacular样例对齐差距说明.md`](Spectacular样例对齐差距说明.md)。
